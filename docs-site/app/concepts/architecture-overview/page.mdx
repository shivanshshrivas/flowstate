# Architecture Overview

`@shivanshshrivas/flowstate` connects four layers to provide crypto-powered e-commerce checkout with escrow and shipping tracking.

## 1. Package Layer (Your Storefront)

The npm package `@shivanshshrivas/flowstate` provides React components and contract ABIs that embed directly in your frontend:

| Export                  | Role                                                                                |
| ----------------------- | ----------------------------------------------------------------------------------- |
| `FlowStateProvider`     | Context provider — injects project config, API key, network, and contract addresses |
| `PayButton`             | One-click checkout — wallet connection, token approval, multi-step checkout overlay |
| `OrderTracker`          | Visual state timeline — shows escrow progress, payout milestones, and XRPL tx links |
| `useFlowState`          | React hook — access resolved config and contract addresses                          |
| `EscrowStateMachineAbi` | ABI for the on-chain escrow contract                                                |
| `MockRLUSDAbi`          | ABI for the FLUSD ERC-20 token contract                                             |

Your server-side code handles webhook signature verification and syncs state updates into your local database.

## 2. Backend API Layer

The FlowState API gateway (Fastify server in `backend/`) orchestrates the full order lifecycle. All routes are under `/api/v1`:

| Group    | Prefix      | Responsibility                                                          |
| -------- | ----------- | ----------------------------------------------------------------------- |
| Auth     | `/auth`     | Project bootstrap, API key creation and rotation                        |
| Orders   | `/orders`   | Create order, select shipping, confirm escrow, advance states, finalize |
| Shipping | `/shipping` | Shippo rate lookup, label purchase, tracking, inbound carrier webhook   |
| Sellers  | `/sellers`  | Onboarding, order queues, metrics, payout history                       |
| Disputes | `/disputes` | Buyer dispute creation, seller response, admin resolution               |
| Platform | `/platform` | Analytics, seller risk views, gas cost estimates                        |
| Webhooks | `/webhooks` | Destination registration and delivery logs                              |
| Agents   | `/agents`   | Role-based AI chat routing via Pinata agents                            |

Supporting infrastructure:

- **PostgreSQL** — direct access via `postgres` driver (no ORM)
- **Redis + BullMQ** — optional async queues for webhook delivery, state transitions, and agent routing
- **WebSocket** — real-time project-scoped event push at `/ws`
- **Cron scheduler** — auto-finalization (every 15 min) and dispute auto-resolve (every 30 min)

## 3. Smart Contract Layer

Three Solidity contracts deployed on **XRPL EVM Sidechain Testnet** (chain ID `1449000`):

**FLUSD** — Mock stablecoin (ERC-20, 6 decimals) simulating RLUSD. Public `faucet()` mints 50,000 FLUSD per call with a 1-hour cooldown.

**EscrowStateMachine** — Core escrow contract implementing a 7-state FSM:

```text
ESCROWED -> LABEL_CREATED -> SHIPPED -> IN_TRANSIT -> DELIVERED -> FINALIZED
                                                          \-> DISPUTED
```

- `createEscrow()` locks buyer tokens and immediately releases the first payout tranche (15%) to the seller
- `advanceState()` moves to the next state and releases the corresponding payout
- `finalize()` releases the holdback minus platform fee after a grace period
- `initiateDispute()` freezes remaining funds

Default payout schedule: `[1500, 1500, 2000, 3500, 1500]` bps (15% / 15% / 20% / 35% / 15%).

**DisputeResolver** — Manages dispute lifecycle. Buyers submit evidence (pinned to IPFS). Sellers have 72 hours to respond. Admin can resolve as refund, release, or custom split. Auto-resolves in buyer's favor if seller misses the deadline.

## Runtime Flow

```text
1. Buyer clicks PayButton
2. CheckoutOverlay collects shipping address and option
3. POST /orders/create -> order in INITIATED state, Shippo rates returned
4. POST /orders/:id/select-shipping -> Shippo label purchased, IPFS pin, USD-to-token conversion
5. POST /orders/:id/confirm-escrow -> on-chain deposit verified, order moves to ESCROWED
6. Seller confirms label printed -> LABEL_CREATED, 15% payout released
7. Shippo webhook: carrier picks up -> SHIPPED, 15% released
8. Shippo webhook: carrier in transit -> IN_TRANSIT, 20% released
9. Shippo webhook: delivered -> DELIVERED, 35% released
10. Grace period expires (48h) -> auto-finalize, holdback (15%) released minus 2.5% platform fee
```

At each step, the backend emits `order.state_changed` webhooks to your registered endpoints and pushes real-time updates over WebSocket.

## Delivery And Eventing

- **Outbound webhooks** — HMAC-SHA256-signed payloads with `X-FlowState-Signature` header
- **Queue mode** (Redis) — BullMQ delivers webhooks with 5 retries and exponential backoff
- **Inline mode** (no Redis) — synchronous in-process dispatch, no queue retries
- **WebSocket** — project-scoped channel at `/ws` with 5-second auth window
- **Cron** — auto-finalize delivered orders past grace period, auto-resolve expired disputes
